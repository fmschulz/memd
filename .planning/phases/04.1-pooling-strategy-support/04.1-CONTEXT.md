# Phase 4.1: Pooling Strategy Support - Context

**Gathered:** 2026-01-30
**Status:** Ready for planning

<domain>
## Phase Boundary

Add architectural support for multiple embedding pooling strategies (mean-pooling and last-token pooling) to enable upgrading from all-MiniLM-L6-v2 (384-dim, mean pooling) to Qwen3-Embedding-0.6B (1024-dim, last-token pooling). This unlocks +15% MTEB score improvement and enables future model flexibility.

**In scope:**
- PoolingStrategy enum (Mean, LastToken variants)
- last_token_pooling() implementation in OnnxEmbedder
- Configurable embedding dimensions (384 or 1024)
- Qwen3 model download and integration
- Backward compatibility with existing mean-pooling models

**Out of scope:**
- New retrieval modes or query types (separate phases)
- Index migration tooling (basic rebuild is sufficient)
- Performance optimization beyond basic implementation
- Multi-model support (single model per daemon instance)

</domain>

<decisions>
## Implementation Decisions

### Model Selection & Migration
- **Default model:** Keep all-MiniLM-L6-v2 as default (preserve existing behavior)
- **Qwen3 opt-in:** Users enable via CLI flag at startup (e.g., `--embedding-model qwen3`)
- **Single model per daemon:** One daemon instance runs one model (simpler architecture)
- **Index migration:** Dimension mismatch triggers error with clear rebuild instructions
  - Error message: "Model changed (384 → 1024 dims). Run with --rebuild-index flag."
  - No automatic rebuild (user explicitly controls)

### Pooling Strategy Architecture
- **Strategy enum:** `PoolingStrategy { Mean, LastToken }` in embedding config
- **Per-model configuration:** Pooling strategy tied to model choice, not user-configurable
  - all-MiniLM-L6-v2 → Mean pooling (existing behavior)
  - Qwen3-Embedding-0.6B → LastToken pooling (new capability)
- **Implementation location:** Add `last_token_pooling()` method to `OnnxEmbedder` struct
- **Backward compatibility:** All existing tests must pass with mean pooling unchanged

### Quality & Performance Targets
- **Success threshold:** 92-95% recall improvement (from 87.5% baseline)
- **Performance acceptable:** p50 latency <50ms (3-4x slower than current 13ms is acceptable for quality gain)
- **Evaluation approach:** Run existing hybrid eval suite with Qwen3, compare to baseline
- **Failure handling:** If Qwen3 underperforms expectations (<90% recall), document findings and keep all-MiniLM as recommended default

### Configuration Defaults
- **Embedding dimension:** Auto-detect from model choice (384 for all-MiniLM, 1024 for Qwen3)
- **HNSW config:** Dimension updated to match embedding model
- **Model URLs:**
  - all-MiniLM: `Xenova/all-MiniLM-L6-v2` (current)
  - Qwen3: `onnx-community/Qwen3-Embedding-0.6B-ONNX` (614MB quantized)

### Claude's Discretion
- Exact implementation of last-token pooling logic (tensor slicing approach)
- Error message wording and formatting
- Test coverage strategy (unit vs integration test balance)
- Logging verbosity for model loading and pooling selection

</decisions>

<specifics>
## Specific Ideas

**From QWEN3_UPGRADE.md:**
- "Use Option 1: Implement Last-Token Pooling" (2-4 hour effort estimate)
- ONNX model from onnx-community repo uses instruction formatting: `Instruct: {task}\nQuery: {text}`
  - May need to add instruction wrapper for Qwen3 queries
- Dimension change 384→1024 requires index rebuild (existing chunks re-indexed)

**Implementation reference:**
```rust
match self.config.pooling {
    PoolingStrategy::Mean => self.mean_pooling(&output, &attention_mask),
    PoolingStrategy::LastToken => self.last_token_pooling(&output),
}
```

**Projected improvements (from MTEB benchmarks):**
- Keyword Recall: 87.5% → 92-95% (+7%)
- Semantic Recall: 87.5% → 92-95% (+7%)
- Mixed Recall: 75.0% → 85-90% (+15%)

</specifics>

<deferred>
## Deferred Ideas

**Migration tooling** — Automatic index migration or dual-index support during transition
- Belongs in future "Index Management" phase
- Current approach: simple rebuild with `--rebuild-index` flag

**Multi-model support** — Different tenants using different embedding models
- Architectural complexity not justified for current use case
- Could be Phase 5.5 or later if needed

**Custom ONNX exports** — Building mean-pooling version of Qwen3
- High effort (4-8 hours), maintenance burden
- Only pursue if last-token pooling proves problematic

**HuggingFace Transformers integration** — Replace ONNX Runtime with native PyTorch
- Defeats purpose of Rust/ONNX (performance, single-binary packaging)
- Out of scope for architecture A

</deferred>
